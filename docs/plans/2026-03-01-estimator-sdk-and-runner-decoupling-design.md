# Estimator SDK and Runner Decoupling Design

Date: 2026-03-01
Status: Approved in discussion

## 1. Goal

Refactor estimator integration so participant code is fully decoupled from evaluator internals, with an installable participant-first CLI and a stable class-based API that supports both local testing and future cloud sandbox execution.

Primary outcomes:

- Participants only need to author `estimator.py` (plus optional `requirements.txt`).
- Estimator lifecycle is stateful and explicit.
- Scoring is separated from estimator execution transport/process details.
- Local and hosted evaluation paths share one execution contract.

## 2. Participant Lifecycle (Target UX)

1. Install CLI tool globally once.
2. Initialize project scaffold.
3. Edit only `estimator.py` for estimator logic.
4. Run local contract validation.
5. Run local scoring in human or agent mode.
6. Package submission artifact.
7. Submit later using same package contract.

Design principle: participant should think only about estimator behavior, not evaluator internals or infrastructure wiring.

## 3. Submission Contract

Required participant files:

- `estimator.py`

Optional participant files:

- `requirements.txt`
- `submission.yaml` (participant-authored metadata)
- `APPROACH.md` (participant-authored long-form writeup)

Cloud-facing artifact (generated by CLI packaging) includes:

- `estimator.py`
- optional `requirements.txt`
- generated `manifest.json`
- optional participant metadata files

## 4. CLI Product Contract

Installable CLI from day 1:

- Package name: `circuit-estimation-cli`
- Canonical command: `cestim`
- Optional long alias: `circuit-estimation`

Core commands:

- `cestim init`
- `cestim validate`
- `cestim run`
- `cestim package`
- `cestim submit` (future)

Output mode rules:

- `--agent-mode`: JSON only, pretty printed, no extra text.
- human mode (default): rich report on success, readable structured error on failure.

## 5. Estimator Interface

Participant estimator class contract:

- must subclass `BaseEstimator`
- required method: `predict(circuit, budget)`
- optional methods: `setup(context)`, `predict_batch(circuits, budget)`, `teardown()`

`predict_batch` default behavior: sequentially call `.predict(...)` and stack outputs.

Setup policy:

- `setup` is hard-capped by timeout/memory limits.
- setup runtime is excluded from score.

Prediction contract:

- `predict` returns `np.ndarray` shape `(circuit.d, circuit.n)`
- float32-compatible numeric values
- finite values only

Batch contract:

- `predict_batch` returns `np.ndarray` shape `(batch, depth, width)`

## 6. Class Discovery and Entrypoint Resolution

Loader rules:

1. Default: class named `Estimator`.
2. Explicit override: `--class ClassName`.
3. If no default and multiple candidates exist: fail with clear guidance.
4. If exactly one candidate exists: auto-select and report selection.

Packaging freezes entrypoint in `manifest.json` so cloud evaluator never relies on runtime autodiscovery.

## 7. Metadata Format Decision

Execution-critical metadata:

- `manifest.json` (generated, machine-validated, deterministic)

Participant-authored metadata:

- `submission.yaml` (descriptive metadata)
- `APPROACH.md` (long-form explanation)

Rationale:

- JSON for strict, security-sensitive execution contract.
- YAML/Markdown for human-authored descriptive content.

## 8. Runner Architecture

Scoring must depend on a runner interface, not direct estimator calls.

Runner implementations:

- `InProcessRunner` for fast local debugging.
- `SubprocessRunner` for realistic local isolation and limits.
- `CloudRunner` later, sharing same logical contract.

Runner lifecycle:

1. `start(entrypoint, context, limits)`
2. `predict(...)` and optional `predict_batch(...)`
3. `close()`

Per-call outcome should include predictions, timings, memory metrics, and status code.

## 9. Scorer Interface Changes

New primary scorer path should consume runner outcomes rather than callable estimator internals.

Key policy behavior:

- setup failures are surfaced as structured run failures.
- predict-time failures are converted into structured per-call failures and scored via existing timeout/fallback policy.
- existing `score_estimator(...)` can be retained temporarily as compatibility wrapper over `InProcessRunner`.

## 10. Error Handling Contract

Structured error model is shared across modes.

- agent mode: JSON payload with stable `stage`, `code`, `message`, optional `details` and `hint`.
- human mode: readable rendering of same structured error.
- tracebacks hidden by default; shown only with `--debug`.
- failures exit non-zero in both modes.

## 11. Migration and Backward Compatibility

Current in-repo reference estimators will be migrated later to `BaseEstimator` implementations.

Refactor sequence:

1. land SDK + loader + runner abstractions
2. keep compatibility path for existing scorer call sites
3. migrate reference estimators
4. remove legacy direct-call integration once all call sites are switched
